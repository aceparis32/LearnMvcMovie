    Working with SQL Server LocalDB

    2017-3-7 3 min to read Contributors
    • all

    In this article

    1. SQL Server Express LocalDB
    2. Seed the database
      3. +

    By Rick Anderson+

    The MvcMovieContext object handles the task of connecting to the database and mapping Movie objects to database records. The database context is registered with the Dependency Injection container in the ConfigureServices method in the Startup.cs file:2

    C#
    public void ConfigureServices(IServiceCollection services)
{
    // Add framework services.
    services.AddMvc();

    services.AddDbContext<MvcMovieContext>(options =>
            options.UseSqlServer(Configuration.GetConnectionString("MvcMovieContext")));
}

    The ASP.NET Core Configuration system reads the ConnectionString. For local development, it gets the connection string from the appsettings.json file:3

    JavaScript
    "ConnectionStrings": {
  "MvcMovieContext": "Server=(localdb)\\mssqllocaldb;Database=MvcMovieContext-2;Trusted_Connection=True;MultipleActiveResultSets=true"
}

    When you deploy the app to a test or production server, you can use an environment variable or another approach to set the connection string to a real SQL Server. See Configuration for more information.+

    SQL Server Express LocalDB

    LocalDB is a lightweight version of the SQL Server Express Database Engine that is targeted for program development. LocalDB starts on demand and runs in user mode, so there is no complex configuration. By default, LocalDB database creates "*.mdf" files in the C:/Users/<user> directory.+

    • From the View menu, open SQL Server Object Explorer (SSOX).

    • Right click on the Movie table > View Designer

    Note the key icon next to ID. By default, EF will make a property named ID the primary key.4

    • Right click on the Movie table > View Data

    Seed the database

    Create a new class named SeedData in the Models folder. Replace the generated code with the following:4

    C#
    using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;
using System;
using System.Linq;

namespace MvcMovie.Models
{
    public static class SeedData
    {
        public static void Initialize(IServiceProvider serviceProvider)
        {
            using (var context = new MvcMovieContext(
                serviceProvider.GetRequiredService<DbContextOptions<MvcMovieContext>>()))
            {
                // Look for any movies.
                if (context.Movie.Any())
                {
                    return;   // DB has been seeded
                }

                context.Movie.AddRange(
                     new Movie
                     {
                         Title = "When Harry Met Sally",
                         ReleaseDate = DateTime.Parse("1989-1-11"),
                         Genre = "Romantic Comedy",
                         Price = 7.99M
                     },

                     new Movie
                     {
                         Title = "Ghostbusters ",
                         ReleaseDate = DateTime.Parse("1984-3-13"),
                         Genre = "Comedy",
                         Price = 8.99M
                     },

                     new Movie
                     {
                         Title = "Ghostbusters 2",
                         ReleaseDate = DateTime.Parse("1986-2-23"),
                         Genre = "Comedy",
                         Price = 9.99M
                     },

                   new Movie
                   {
                       Title = "Rio Bravo",
                       ReleaseDate = DateTime.Parse("1959-4-15"),
                       Genre = "Western",
                       Price = 3.99M
                   }
                );
                context.SaveChanges();
            }
        }
    }
}

    If there are any movies in the DB, the seed initializer returns and no movies are added.4

    C#
    if (context.Movie.Any())
{
    return;   // DB has been seeded.
}

    Add the seed initializer

    • ASP.NET Core 2.x
    • ASP.NET Core 1.x
      • +

    Add the seed initializer to the Main method in the Program.cs file:6

    C#
    using Microsoft.AspNetCore;
using Microsoft.AspNetCore.Hosting;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Logging;
using MvcMovie.Models;
using System;

namespace MvcMovie
{
    public class Program
    {
        public static void Main(string[] args)
        {
            var host = BuildWebHost(args);

            using (var scope = host.Services.CreateScope())
            {
                var services = scope.ServiceProvider;

                try
                {
                    // Requires using RazorPagesMovie.Models;
                    SeedData.Initialize(services);
                }
                catch (Exception ex)
                {
                    var logger = services.GetRequiredService<ILogger<Program>>();
                    logger.LogError(ex, "An error occurred seeding the DB.");
                }
            }

            host.Run();
        }

        public static IWebHost BuildWebHost(string[] args) =>
            WebHost.CreateDefaultBuilder(args)
                .UseStartup<Startup>()
                .Build();
    }
}

    Add the seed initializer to the end of the Configure method in the Startup.cs file.+

    C#
                app.UseStaticFiles();
            app.UseMvc(routes =>
            {
                routes.MapRoute(
                    name: "default",
                    template: "{controller=Home}/{action=Index}/{id?}");
            });

            SeedData.Initialize(app.ApplicationServices);
        }
    }
}

    @muyiwataiwoThat syntax is needed only for the purpose of seeding the database and running entity framework migrations when the application is starting up. It is not generated for you so you would need to write code like that yourself if you need to seed your database and run entity framework migrations during startup of your own application.


    The main points to understand is that DI (Dependency Injection), manages creating services for you such as DbContext, and it manages the lifetime of the service too. Normally in an asp.net core application any DbContext services get registered as "scoped" dependencies which means they are created and injected as needed per web request and then they are automatically disposed for you at the end of the web request. So for normal use you would just add a DbContext to the constructor signature of a controller or add another service to the controller constructor that would then have a its own dependency on the DbContext, whatever you add to the constructor signature would be injected for you. In that case you don't need to wrap any "using" statement around DbContext and you don't need to dispose it when you are finished, that is done for you automatically by the DI at the end of the web request. DI can handle creating all the dependencies through all the layers of your application. So a controller could have a dependency on a service that has its own dependency on a DbContext and DI can create both the DbContext and the serivce that wraps it and inject that into your controller, or you could have a DbContext injected directly in your controller.


    But things are different if you need to access a DbContext during the startup of the program. After application startup the application starts listening for web requests but during startup there is no web request so there is no automatic scoping based on a web request at that point. So in that case you have to manually create a scoped service provider to get the DbContext for you. By wrapping that in a "using" statement it ensures that the scope and and scoped services you use inside the using statement get disposed when you are finished with them.

    Opening SSOX revealed a '(localdb)\ProjectsV13' but there was no sign of the DB (it did exist on disk in the noted location).  I clicked the 'Add SQL Server' button in the SSOX window, selected 'Local', and then the '(localdb)\MSSQLLocalDB' option was listed.  Having selected that and Ok'd the dialog, the project database became accessible in the tree.

    Many Thanks for this tutorial :)
